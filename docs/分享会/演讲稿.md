# 从提示词工程到驾驭工程，Harness 框架的五个问题

**完整演讲稿（含 3 组互动）**


## PPT 01

大家平时用 CodeX 吧，大家有没有想过一个问题。

CodeX 用的模型，跟你在网页版里用的模型其实是同一个。但体验完全不一样。

网页版里，你只能跟它聊天。CodeX 能读你的项目结构、帮你改代码、跑测试。

同样的模型，为什么能干的事完全不同？

Anthropic 说过一句话，Claude 本身只是一部分，真正让它能干活的是外面的工具和基础设施。

今天聊的，就是这套系统。


## PPT 02

在具体聊这套系统之前，我们先看一个演进的过程。

最早大家接触 LLM 的时候，我们学的是提示词工程。

系统提示词怎么设计，思维链怎么引导，教的是模型"怎么回答"。

后来发现光靠提示词不够。

模型不知道你的数据，就有了 RAG。

模型记不住历史，就有了 Memory。

模型没法调用外部系统，就有了 Function Calling。

这些东西本质上是在做一件事：把模型需要的信息、能力、约束全部塞进上下文。

这就是上下文工程。

但上下文工程也有天花板。当你真正要做生产级 Agent 的时候，你会发现还要解决另一类问题。

文件怎么安全地读写？任务做到一半断了怎么恢复？子任务怎么隔离？上下文超了怎么办？执行过程怎么观测？

这些问题跟提示词没关系，跟上下文也没关系，是系统架构的问题。

从提示词工程到上下文工程，再到驾驭工程，这是一个层层递进的过程。

提示词工程教模型怎么回答。上下文工程给模型可用的东西。驾驭工程管的是 Agent 这个系统怎么跑起来、怎么跑得稳、怎么跑得安全。

Harness 框架就是驾驭工程在架构层面的一种落地方式。


## PPT 03

那为什么这套系统成了刚需？没有 Harness 框架时，做 Agent 到底是什么状态？

假设你要做一个编程助手，得从零开始写 Agent 的文件读写、任务管理、子任务调度、上下文管理。这些东西加起来可能上万行的代码，跟业务逻辑没关系。每个项目都要重复一遍，而且每个项目写的还不一样。

这跟早年做 Web 开发一样，所有东西都得自己从零搭，路由自己写、数据库连接自己管、请求参数自己解析。每个项目都把这些重复一遍，写法还不一样，bug 也千奇百怪。后来框架出来了，没人再这么干了。Agent 开发现在就在这个路口。


**【互动问答 ②】**

> 在座有没有经历过那个年代的，所有底层东西都得自己写，框架还没出现，每开一个项目就是一次重复劳动？
>
> 现在的 Agent 开发，其实处在同样的位置。Harness 要做的，就是把这一层标准化。


## PPT 04

Harness 框架在生态里什么位置？以 LangChain 生态为例，分三层。

最底层是 LangGraph，负责 Agent 的运行时。比如 Agent 跑到一半服务器挂了，能从断点恢复。用户想看实时进度，能流式输出。这些都由 LangGraph 提供。

中间层是 LangChain，提供模型抽象、工具接口、ReAct 循环。你写一个 Python 函数就能变成 Agent 可调用的工具。

最上层是 Deep Agents，也就是 Harness 套件层。模型以外的东西都归它管，文件系统、任务规划、子任务分配、长期记忆。

打个比方，LangGraph 是工作台，LangChain 是工具箱，Deep Agents 是预制好的工作间。


## PPT 05

在讲 Harness 框架具体解决了什么之前，先看 Agent 最原始的样子。

Agent 的核心就是一个循环。循环分三步。

第一步，定义工具。告诉模型工具叫什么、干什么用、需要什么参数。第二步，把工具注册给 Agent。第三步，进入主循环。

用户说"帮我生成一张赛博朋克风格的猫"。模型看到这个请求，再看看手头可用的工具，决定调用 generate_image。

注意这里的关键点。模型返回的不是一段话，而是一个结构化的 JSON。里面包含了要调用的工具名和参数。这是 Function Calling 的核心机制。模型负责决定要调什么，但系统负责解析和执行。

系统拿到这个 JSON 之后，解析出工具名和参数，找到对应的工具函数真正去执行。执行完拿到图片链接，把工具调用和结果都记入消息历史。

下一轮循环模型看到这些信息，就知道工具已经执行完了，结果是图片链接，于是直接回答用户"图片生成成功"。循环结束。


## PPT 06

这个原始形态看起来很清晰，对吧？但真要把它做成生产级 Agent，你会发现一系列问题。

这些问题跟 Agent 的推理能力没关系，纯粹是系统设计的问题。

第一，文件怎么安全地操作？命令怎么安全地执行？每个项目都在自己写 read_file、write_file、execute，而且 execute 这个工具可以直接任意命令，没人拦着。

第二，任务做到一半断了，怎么恢复进度？谁帮它记住"我做到哪了"？

第三，子任务全部挤在一个上下文里跑，窗口扛不住怎么办？

第四，对话越来越长，Token 超了怎么办？

第五，Agent 里面发生了什么，完全看不到，出了 bug 怎么调试？

每个做 Agent 的项目都在自己解决这些问题，而且解决的方式千奇百怪。这就是 Harness 框架要填的坑。


## PPT 07

第一个问题，Agent 怎么安全地操作文件？

没有 Harness 框架的时候，execute 是自己实现的。Agent 拿到之后，他想干什么都行。rm -rf / 删光文件，网络拉取木马，你管不住，也拦不住他。

那是不是干脆不给 execute？也不行。

以我们项目为例，我们项目需要 PDF Skill，PDF Skill 内置 Python 脚本。Agent 得先 pip install，再跑脚本，才能干活。装依赖要 execute，跑脚本也要 execute。不给 execute，Skill 就是摆设。

所以问题不是"给不给"，是"怎么安全地给"。


**【互动问答 ③】**

> 说到这儿，我想问大家一个问题——
>
> 如果你现在要做一个给团队用的编程助手，你敢直接把 execute 权限给 Agent 吗？
>
> 可能有人觉得"敢，我盯着它跑"，有人觉得"不敢，太吓人了"，也有人会说"给它加一层容器再跑"。
>
> 选容器的朋友，跟 Harness 的思路是一致的——解法不是把 execute 砍掉，而是把 execute 关进沙箱里。

Harness 的解法分两层。

第一层，虚拟文件系统。把 glob、read_file、write_file 这些都封装成标准工具，后端可以换——放内存、放磁盘、放持久化存储都行。但如果选 local 要小心，就算不给 execute，Agent 照样能读 .env、读私钥、读源码。

第二层，沙箱。配了沙箱之后，所有操作都在里面跑，出不去。想删文件？删的是沙箱自己的。想读敏感信息？沙箱里压根没有。

## PPT 09

第二个问题，Agent 怎么记住做到哪了？

没有 Harness 框架的时候，传统做法是用 JSON 文件存状态。但 JSON 只能存当前状态，存不了状态变化的历史。一个任务从待执行变成执行中，再变成已完成，中间的过程全丢了。万一执行到一半中断了，你只知道它最后是"执行中"，不知道具体卡在哪一步。

Harness 框架的解法是，调用了一个 write_todos 工具，把任务列表写进运行时状态。每个任务有 "待执行" "执行中" "已完成" 三种状态，每完成一步就更新一次。关键是这个任务列表跟对话历史分开存，每次更新都是增量追加。就算执行到一半断了，一查状态就知道做到哪一步了，直接恢复。

## PPT 11

第三个问题，子任务太重怎么办？

Agent 碰到一个特别重的子任务，如果跟主任务挤在同一个上下文里跑，窗口迟早扛不住。

没有 Harness 的时候，大家自己写调度逻辑，写法各不相同，但问题都一样——上下文隔离做不好，主 Agent 动不动就被撑爆。

Harness 的解法是主 Agent 调用 task 工具，系统创建一个全新的子 Agent 实例，跟主 Agent 完全隔离。子 Agent 跑完之后只把最终结果返回给主 Agent，中间所有的操作和步骤都留在自己的上下文里，主 Agent 这边干干净净。


## PPT 12

主 Agent 调用 task 委派任务。子 Agent 执行，返回结果。主 Agent 只收到结果。


## PPT 13

第四个问题，Token 超限怎么办？

Agent 跑久了，对话越来越长，工具返回结果越来越大，Token 迟早会超。

没有 Harness 框架的时候，自己想办法。要么写摘要，但会丢信息。要么手动截断，但会破坏上下文。要么直接重启会话，但会丢状态。每种方案都有代价。

Harness 框架的解法是上下文卸载。大文件内容不直接塞进对话历史，而是放在虚拟文件系统里，对话历史只存一个指针。Agent 需要读的时候，按偏移量分段读取，用多少取多少。

对话历史里只有指针，真正的数据按需获取。Token 不会爆，信息也不会丢。

这就是 Harness 的答案。上下文不用自己管，框架在后台帮你卸掉了。

## PPT 15

第五个问题，执行过程是黑盒怎么办？

前四个问题都是让 Agent 能干活。第五个问题，怎么知道它干得对不对？


**【互动问答 ④】**

> 问一个特别现实的问题——
>
> 有没有遇到过这种情况：Agent 返回了"成功"，耗时也正常，Token 也没超，但最后给的结果是错的？
>
> 这就是这类系统最棘手的地方，它看起来很对，但结论是错的。
>
> 传统监控发现不了，因为你监控的是 CPU 和延迟，不是"推理对不对"。所以我们需要新的观测手段。


没有 Langfuse 的时候，你只能看到输入和输出。中间发生了什么，完全不知道。

问题是中间每一步都很关键。路径动态变化，没法提前埋点。成本不透明，钱花在哪不知道。Agent 兜圈子浪费 Token，你发现不了。最麻烦的是，它返回成功、延时正常，但结论是错的。幻觉，传统监控发现不了。

Langfuse 从三个角度解决这个问题。

第一个是链路追踪。宏观上按用户看、按会话看。发现异常之后钻到具体的 Trace，看到每一步调了什么工具、花了多少钱。

第二个是质量评估。用另一个大模型来打分，完整性对不对、推理对不对、工具调用合不合理，每个维度都评。对结果有疑问的时候，人工可以兜底。

第三个是提示词管理。以前改提示词要走研发流程、等发版，天级别甚至周级别。现在 PM 自己就能改，分钟级生效。

三个角度合在一起，执行过程从黑盒变成白盒。出了 bug 不用猜，打开链路一看就知道问题出在哪。

## PPT 17

好，五个问题都讲完了。最后用一个完整案例串一下。

用户说"帮我写一篇小红书笔记，介绍今天吃的这家餐厅，然后发布"。这个任务有两件事要做，写笔记和发笔记。

怎么写？通过 Skill 来解决。Skill 是打包好的领域知识，比如小红书笔记的写作风格、标题怎么起、正文怎么组织、标签怎么加。Skill 有一个特点叫渐进式加载，启动的时候只读名字和描述，真正需要的时候才加载全部内容，不占用上下文窗口。Agent 看到"小红书"这个关键词，加载对应的 Skill，按照规范生成文案。

怎么发？通过 MCP 工具来解决。MCP Server 暴露了 post_to_xhs 这个工具，Agent 直接调用发布。Agent 只知道调了一个工具，不知道背后是谁提供的。这就是解耦。

整个流程是这样的。主 Agent 收到任务之后，判断需要小红书写作知识，加载 Skill 生成文案，然后调用 MCP 工具发布出去。整个过程用户只看到最终结果，背后 Skill 管怎么写，MCP 管怎么发，各司其职。
